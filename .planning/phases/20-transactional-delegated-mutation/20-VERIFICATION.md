### CLASS B — fs_read_deny is INERT against AppContainer children (RESOLVED — real production defect, NOT a regression)
**VERDICT (hardware-proven, investigator a2b1920):** The 3 tests assert a guarantee Windows does not provide. The package-SID DENY ace IS applied and canonically ordered (DENY idx0 before inherited ALLOW idx1, same SID) — proven via a standalone DACL dump. But **Windows' AppContainer lowbox access check does NOT honor a DENY ACE against the AppContainer's OWN package SID** — proven by a minimal AppContainer (SECURITY_CAPABILITIES only, zero sandbox token code) reading straight through the canonical DENY→ALLOW DACL (exit 0, secret disclosed). Therefore: NOT a DACL-ordering bug, NOT a security regression from the deny-only-SID removal (reproduces with all normal SIDs enabled), ORTHOGONAL to core#254 #2. `normal_sid_only_grant_is_denied` PASSING is consistent — AppContainer denial comes from WITHHOLDING the package grant, never from a package DENY.
**Real defect exposed:** `fs_read_deny` is implemented as a package-SID DENY ace (acl_lease.rs:425 `IntentKind::Deny` → :527 DENY_ACCESS → :839 explicit_access_for_sid) which is INERT → `fs_read_deny` currently provides NO protection for the deny-a-subpath-under-a-granted-tree case.
**FIX (proven on hardware — exit 1 "Access is denied."):** For `IntentKind::Deny`, stop adding the package DENY ace; enforce deny by REMOVING the grant path — set the target DACL `PROTECTED_DACL_SECURITY_INFORMATION` (0x80000000) and strip every package-SID ALLOW (incl. the inherited one), leaving only normal SIDs (AppContainer ignores them for granting). Touches `apply_intents` (521-537), `explicit_access_for_sid` (839-852), `apply_explicit_access` (854-883); cleanup (`cleanup_locked`/`remove_and_verify_exact_sid`) must clear PROTECTED / restore inheritance on revoke. Directory deny targets protected per-object. This makes fs_read_deny a REAL control AND makes the 3 tests pass asserting real denial. Repro artifacts: C:\aclrepro on box, scratchpad/aclrepro/ local.

### CLASS C — timeout test-hold artifact (RESOLVED — test bug, timeout path is correct)
**VERDICT (hardware-proven, investigator ac6919d):** The 150ms timeout path is CORRECT. `echo_temp_and_hold(8)` uses `choice.exe /T 8 /D Y` which **exits instantly (~26ms) under the sandbox's NULL stdin** (proven: choice /T 4 <NUL = 26ms; ping/waitfor/Start-Sleep all held ~4s; choice WITH a real pipe stdin blocked 135s). So the child finishes in ~26-67ms < 150ms → WAIT_OBJECT_0 → `Ok`, never reaching `if timed_out { Err(Timeout) }` (process.rs:960). The in-source comment (live_fs_acl.rs:120-121) claiming "choice tolerates the sandbox's null stdin" is factually wrong on this hardware. `clamp_timeout_ms` (handles.rs:287) has no floor — 150ms passes through, path is sound.
**CROSS-CUTTING:** the broken hold affects EVERY test using `type_and_hold`/`echo_temp_and_hold` (timing-dependent present-during-run waits) — may also feed Class D.
**FIX:** replace the `choice` hold with `waitfor.exe /t {seconds} wlhold >nul 2>&1 & exit /b 0` (stdin/console/network-independent; verified present on box) in `echo_temp_and_hold` (line 143) and `type_and_hold` (line 132). NOT `ping 127.0.0.1` — AppContainer built with zero capabilities (process.rs:544-545), loopback is network-isolated. Fix/delete the wrong comments at live_fs_acl.rs:120-130,138-141.
