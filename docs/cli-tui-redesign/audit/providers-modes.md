## Modes & approval — the honest depiction

The mockup conflates **three distinct engine concepts** under one "mode" badge. They are orthogonal and the redesign should keep them separate.

**1. Approval mode (`SessionMode`)** — the real, switchable thing. Three values only:
`wcore-protocol/src/commands.rs:79-85`
- `Default` — every tool call needs approval (nothing auto-approved). `is_auto_approved` returns false for all categories (`lib.rs:101-105`).
- `AutoEdit` — auto-approves `info` + `edit` categories; still prompts for `exec` (Bash) and `mcp` (`lib.rs:103`).
- `Yolo` — auto-approves everything (`lib.rs:102`).

`current_mode()` strings are `default / auto_edit / yolo` (`lib.rs:127-137`). The badge in the status bar should show one of these three — **humanized as `Default` / `Auto-edit` / `Yolo`**, not "Ready" and not "Plan". "Ready" is a connection/idle state (a fine thing to show — just don't call it a mode). If the redesign wants a `⇧Tab` cycle, the cycle is `Default → Auto-edit → Yolo` and it must drive the existing `SetMode` command. Today there is no keybinding and no cycle — `--auto-approve` is a one-shot boolean flag (`main.rs:138-140`).

**2. Plan mode** — real but architecturally different. It is **not** a `SessionMode`. It is a pair of tools (`EnterPlanModeTool` / `ExitPlanModeTool`, `plan/tools.rs:21,102`) the *model itself* invokes; while active the engine restricts the tool set to read-only (`tools.rs:18-20`). So the mockup's central Plan-Review claim — "proposes a plan, writes nothing until you approve" — is **genuinely backed by the engine**. What is invented:
- Entry via `Shift+Tab` — the engine has no such trigger. Plan mode is entered when the agent calls `EnterPlanMode` (or a skill requests it). A TUI affordance to *suggest* plan mode is reasonable net-new UI, but it must ultimately invoke that tool, not a fake mode toggle.
- The buttons `Approve & run / Edit in $EDITOR / Reject`. The real exit is `ExitPlanMode`; user sign-off on the plan flows through the generic approval path (`ApprovalScope`, `ApprovalResume`). There is no `$EDITOR` plan-editing hook in the engine. Showing three buttons is fine as a *redesign*, but label honestly: "Approve" → exit plan mode + proceed; "Reject" → stay in plan mode / redirect. Drop "Edit in $EDITOR" unless that capability is actually built.

**3. Tool approval (`ApprovalScope`)** — this is the per-call modal, and the mockup gets it **right**. `Once` vs `Always`, where `Always` is **category-scoped** (`edit`, `exec`, `info`, `mcp`) and lives for the session (`lib.rs:78-79,96-117`). The 3-option modal ("Apply / Apply + don't ask again for Edit / No") maps 1:1 to `ToolApprove{Once}` / `ToolApprove{Always}` / `ToolDeny{reason}`. Keep this verbatim — it is the most accurate surface in the mockup.

**Bottom line for the redesign:** the status-bar badge should read one of `Default / Auto-edit / Yolo`. Plan mode is a transient overlay state, not a badge value — if shown, show it as a banner ("Plan mode — read-only") distinct from the approval-mode badge. The per-call approval modal is accurate and should be preserved. The `⇧Tab` cycle and `$EDITOR` plan editing are net-new UI proposals, not existing behavior — label them as such in the spec so they get built rather than assumed.

## Provider & model naming notes

Real providers: **Anthropic, OpenAI, AWS Bedrock, Google Vertex** (first-party `LlmProvider` impls in `wcore-providers/src/`: `anthropic.rs`, `openai.rs`, `bedrock.rs`, `vertex.rs`, plus `gemini.rs`). **Ollama is a plugin** (`crates/wayland-ollama`, registers via `register_providers`; `main.rs:65`). The onboarding screen listing Ollama at `localhost:11434` is consistent with that plugin.

Model display: the engine stores raw model strings; `provider_label` is the *requested* alias name, not the underlying type (`wcore-config/src/config.rs:465,664`; test `:2176`). So a status line *could* show `anthropic / claude-sonnet-4-6`. A humanized `claude · sonnet 4-6` is a reasonable TUI render — but the version token is **`4-6`**. Every `4.6` in the mockup (status bar ×4, two AI-turn subtitles) is wrong and must become `4-6`.
