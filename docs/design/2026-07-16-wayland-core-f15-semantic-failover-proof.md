Status: implemented and sealed by the commit containing this document.

Base: `906287e1790ab2e0c8a6f1f71940e9acc2b55c75`

Board crosswalk: FerroxLabs/wayland#692. The issue remains open for release

## Production path

The ordinary engine bootstrap now constructs one `ResilientProvider` around
the active provider and every independently resolved configured fallback. Each
candidate has its own provider/model identity, credentials, compatibility
profile, organization, region, pricing evidence, and cooldown tracker. Runtime
provider/profile rebinding uses the same construction path. The JSON protocol
sink emits a typed failover receipt for every fallback decision.

Live pricing refresh is explicit operator policy. Setting
`WAYLAND_PRICING_AUTO_REFRESH=1` permits a bounded OpenRouter catalog refresh;
a fresh cache avoids network work. Disabled, missing, stale, corrupt, failed,
or unpublished live data falls back to the trusted configured or bundled
catalog and is never represented as fresh live evidence.

## Fail-closed decisions

- Project configuration cannot replace the global provider/organization/
  region/pricing routing floor.
- Tools, vision, structured-output, context-window, provider, organization,
  region, pricing-freshness, pricing-availability, cooldown, and budget checks
  happen before a fallback provider is called.
- A context overflow may route only to a candidate with a known sufficient
  context window. A malformed/format request and context overflow do not poison
  provider health.
- Transient failures use deterministic exponential cooldown with server
  `Retry-After` as a lower bound and exactly one half-open probe. Permanent
  credential, billing, session, and model failures require explicit recovery.
- A configured fallback gets a new budget reservation immediately before its
  physical dispatch. The typed receipt carries the authoritative admission
  estimate, selection, rejection, failure/cooldown reason, and retry time.
- Failover occurs only before response-stream output. Core does not replay a
  request to another provider after partial output.

## Desktop contract

Desktop contract `wayland-desktop-core` is version `1.3`, generated by
`wcore-desktop-contract-gen/4`. Capability
`semantic_failover_receipts=available` pins subcontract `1.0` and the
`provider_failover_receipt` event. Its schema covers successful and rejected
candidates, all reason/rejection variants, nullable evidence, pricing, and
selected-provider identity. Corpus generation, byte comparison, and malformed
authority tests prevent silent wire drift.

## Audit corrections caught before seal

The proof loop found and fixed three issues before commit:

1. The first generated receipt schema inferred only the success fixture and
   therefore could not describe rejected candidates or non-null reasons.
2. Protocol inventory assertions still pinned version `1.2` and 44 events.
3. Format failures were accidentally allowed to poison provider health when
   cooldown classification was reused as circuit-breaker classification.

## Remaining boundaries

- `LlmRequest` currently has no response-schema/structured-output request
  field. Candidate admission has a distinct enforced structured-output
  requirement and provider capability, but the ordinary production request
  builder cannot yet set that requirement.
- macOS and Windows native behavioral execution was not performed in this
  lane. The policy, receipt, config, and pricing logic is platform-neutral;
  authoritative execution evidence here is Linux.
- The opt-in live catalog proves bounded fetch/cache/fallback behavior. It does
  not make an external catalog an authority over provider admission; the
  global routing floor and budget authority remain local.
